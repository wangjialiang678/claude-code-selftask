## 1. 概述与定位

### 1.1 claude-code-selftask（自动任务系统）

| 维度 | 描述 |
|------|------|
| **一句话定位** | 多 Claude Code 实例并行任务执行系统 |
| **灵感来源** | 胡渊鸣《我给 10 个 Claude Code 打工》 |
| **核心问题** | 如何让多个 Claude Code 实例高效并行工作？ |
| **关键隐喻** | **工厂流水线** — Manager 是工头，Worker 是工人，Task Queue 是工单池 |
| **项目阶段** | 已完成 MVP，8/8 测试任务成功执行 |

**核心思路**：把一堆任务放进 JSON 队列，启动多个 Claude Code 进程（每个在独立 Git Worktree 中），让它们自行领取、执行、提交任务。外部 Manager 负责进程生命周期管理。

### 1.2 长时间运行智能体

| 维度 | 描述 |
|------|------|
| **一句话定位** | 跨上下文窗口的自主编码 Agent，带自我改进能力 |
| **灵感来源** | Anthropic 官方《Effective Harnesses for Long-Running Agents》 |
| **核心问题** | 如何让 AI Agent 在有限上下文窗口下完成复杂项目？ |
| **关键隐喻** | **接力赛** — 每个 Session 是一个选手，通过 feature_list + progress + git 传递接力棒 |
| **项目阶段** | 已完成框架 + Observer 系统，经过 4 Session 6 小时实战验证 |

**核心思路**：将复杂项目拆成 feature list，通过多个 Session 接力完成。Session 1 初始化项目结构和特性列表，Session 2+ 依次实现特性。Observer 系统在后台自动收集信号、匹配模式、注入告警，实现自我改进。

### 1.3 自测试迭代系统（sim-dev）

| 维度 | 描述 |
|------|------|
| **一句话定位** | 模拟优先的自动化质量测试 CLI 工具 |
| **灵感来源** | 交互式 AI 系统的快速迭代需求 |
| **核心问题** | 如何在没有真实用户的情况下快速验证 AI 系统质量？ |
| **关键隐喻** | **实验室** — 用模拟用户（虚拟小白鼠）批量测试系统 |
| **项目阶段** | v0.1.0 完成，带完整 CLI、测试套件和报告系统 |

**核心思路**：定义多种用户画像（Persona）和测试场景（Scenario），让模拟用户批量与目标系统交互，通过 Hard Gate（功能正确性）和 Soft Gate（体验质量）双门评估，生成结构化报告和版本对比。

## 2. 核心架构对比

### 2.1 claude-code-selftask 架构

```
┌─────────────────────────────────────┐
│         Web Dashboard (PWA)         │  ← 用户交互层
│    FastAPI + WebSocket + Voice      │
├─────────────────────────────────────┤
│        Worker Manager               │  ← 编排层
│   ┌─────┐ ┌─────┐ ┌─────┐         │
│   │ W-1 │ │ W-2 │ │ W-3 │ ...     │  ← 执行层
│   └──┬──┘ └──┬──┘ └──┬──┘         │
│      │       │       │             │
│   ┌──▼──┐ ┌──▼──┐ ┌──▼──┐         │
│   │ WT1 │ │ WT2 │ │ WT3 │         │  ← 隔离层 (Git Worktree)
│   └─────┘ └─────┘ └─────┘         │
├─────────────────────────────────────┤
│     data/dev-tasks.json             │  ← 状态层 (共享任务队列)
└─────────────────────────────────────┘
```

**架构风格**：Master-Worker 并行架构
- **编排方式**：外部 Manager 管理 Worker 生命周期
- **隔离方式**：Git Worktree（每个 Worker 独立工作目录）
- **通信方式**：共享 JSON 文件 + 原子操作
- **状态管理**：集中式（单一 JSON 文件）

### 2.2 长时间运行智能体架构

```
┌──────────────────────────────────────────┐
│              main.py (Supervisor)         │  ← 监督层
│  ┌────────┐  ┌────────┐  ┌────────┐     │
│  │Session1│→│Session2│→│Session3│→...   │  ← Session 接力
│  │ Init   │  │ Code   │  │ Code   │     │
│  └───┬────┘  └───┬────┘  └───┬────┘     │
│      │           │           │           │
│  ┌───▼───────────▼───────────▼───┐       │
│  │  feature_list + progress + git │       │  ← 上下文传递层
│  └───────────────────────────────┘       │
├──────────────────────────────────────────┤
│  Observer System (Hook / Skill)          │  ← 自我改进层
│  ┌──────┐ ┌────────┐ ┌────────────┐     │
│  │Signal│→│Pattern │→│Auto-fix /  │     │
│  │Collect│ │Matching│ │Alert Inject│     │
│  └──────┘ └────────┘ └────────────┘     │
├──────────────────────────────────────────┤
│  Knowledge Base (patterns/fixes/history) │  ← 知识积累层
└──────────────────────────────────────────┘
```

**架构风格**：Session 接力 + 自我改进环
- **编排方式**：单一 Supervisor 管理 Session 序列
- **隔离方式**：每个 Session 是独立上下文窗口
- **通信方式**：文件系统（feature_list.json, claude-progress.txt, git log）
- **状态管理**：分布式（多种文件各司其职）

### 2.3 自测试迭代系统架构

```
┌──────────────────────────────────────────┐
│           sim-dev CLI                     │  ← 用户接口层
│  init / run / report / compare / serve   │
├──────────────────────────────────────────┤
│       Simulation Engine                   │  ← 模拟层
│  ┌────────┐     ┌──────────┐             │
│  │Persona │  ×  │ Scenario │  = Sessions │
│  │(3种)   │     │(5种)     │   (15组)    │
│  └────────┘     └──────────┘             │
├──────────────────────────────────────────┤
│       Target Adapters                     │  ← 适配层
│  Mock │ HTTP │ Python │ CLI              │
├──────────────────────────────────────────┤
│  ┌──────────┐    ┌──────────┐            │
│  │Hard Gate │    │Soft Gate │            │  ← 评估层
│  │(断言插件)│    │(体验评分)│            │
│  └──────────┘    └──────────┘            │
├──────────────────────────────────────────┤
│  records/ → reports/ → compare/          │  ← 分析层
└──────────────────────────────────────────┘
```

**架构风格**：批处理管道 + 插件化评估
- **编排方式**：CLI 命令驱动，顺序执行
- **隔离方式**：每个 Session 是独立交互序列
- **通信方式**：YAML 配置 + JSON 记录
- **状态管理**：基于文件的记录和报告

### 2.4 架构对比总结

| 维度 | selftask | 长时间运行 | 自测试迭代 |
|------|----------|-----------|-----------|
| **架构模式** | Master-Worker | Session 接力 | 批处理管道 |
| **并发模型** | 多进程并行 | 单 Agent 串行 | 单进程顺序 |
| **状态存储** | 集中 JSON | 分散文件系统 | 分离记录 + 报告 |
| **隔离机制** | Git Worktree | 上下文窗口 | Session 对象 |
| **扩展方向** | 水平（加 Worker） | 纵深（加 Session） | 维度（加 Persona/Scenario） |
| **复杂度** | 中等 | 高 | 中低 |
| **耦合度** | 松耦合（JSON 共享） | 中耦合（文件传递） | 松耦合（适配器模式） |

## 4. Agent 编排模式对比

### 4.1 selftask：并行工厂模式

```
WorkerManager
    │
    ├── Worker-1 ──→ claim task ──→ execute ──→ commit ──→ next task
    ├── Worker-2 ──→ claim task ──→ execute ──→ commit ──→ next task
    └── Worker-3 ──→ claim task ──→ execute ──→ commit ──→ next task
         ↕              ↕              ↕
    ┌────────────────────────────────────┐
    │      Shared Task Queue (JSON)       │
    └────────────────────────────────────┘
```

**特点**：
- **Ralph Loop**：每个 Worker 循环执行 → 领取 → 执行 → 提交 → 退出 → 重启
- **无抢占**：先到先得，原子 claim 防止重复领取
- **无依赖**：任务之间完全独立，不存在前置条件
- **Worker 无状态**：每次重启是全新的 Claude Code 进程

**优势**：吞吐量高，适合大量独立简单任务
**劣势**：不支持任务依赖、不适合复杂多步骤任务

### 4.2 长时间运行：Session 接力模式

```
Session 1 (Init)
    │  ← 读取 app_spec.txt
    │  → 生成 feature_list.json + init.sh
    │  → git commit
    ▼
Session 2 (Code)
    │  ← 读取 feature_list + progress + git log
    │  → 恢复上下文
    │  → 实现 Feature A
    │  → 浏览器测试
    │  → 标记 "passes": true
    │  → git commit
    ▼
Session 3 (Code)
    │  ← 读取 feature_list + progress + git log
    │  → 恢复上下文
    │  → 验证 Feature A 未退化
    │  → 实现 Feature B
    ...
```

**特点**：
- **双角色分工**：Initializer Agent（搭建） vs Coding Agent（实现）
- **上下文恢复**：每个新 Session 通过文件系统重建状态
- **Feature List 不可变性**：只允许改 `passes` 字段，描述只增不改
- **90 分钟超时保护**：防止 Session 无限阻塞

**优势**：能完成大型复杂项目，上下文可跨 Session 传递
**劣势**：串行执行，速度受单 Agent 能力限制

### 4.3 自测试迭代：批量模拟模式

```
sim-dev run
    │
    ├── (novice × basic_task)   → Session → Record
    ├── (novice × edge_case)    → Session → Record
    ├── (expert × basic_task)   → Session → Record
    ├── (expert × edge_case)    → Session → Record
    ├── (impatient × basic_task) → Session → Record
    └── (impatient × edge_case)  → Session → Record
         │
         ▼
    Aggregate → RunSummary → Report
```

**特点**：
- **Persona × Scenario 笛卡尔积**：自动组合所有测试维度
- **SimulatedUser 状态机**：模拟用户的耐心、困惑、放弃行为
- **双门评估**：Hard Gate（必须通过）+ Soft Gate（质量指标）
- **版本对比**：`compare` 命令追踪改进效果

**优势**：覆盖面广，自动化程度高，输出结构化
**劣势**：模拟用户不如真实用户，当前版本不调用 LLM 做评估

### 4.4 编排模式对比总结

| 维度 | selftask | 长时间运行 | 自测试迭代 |
|------|----------|-----------|-----------|
| **Agent 数量** | 多个（1-5+ Worker） | 单个（多 Session） | 无 Agent（模拟用户） |
| **执行方式** | 并行 | 串行 | 串行（可并行化） |
| **任务粒度** | 小而独立 | 大而复杂 | 交互 Session |
| **任务来源** | 外部队列 | Feature List 内部生成 | Persona × Scenario 组合 |
| **失败处理** | 标记 failed，释放任务 | 超时重启新 Session | 记录为断言失败 |
| **扩展方式** | 加 Worker | 加 Session | 加 Persona/Scenario |

## 5. 任务管理机制对比

### 5.1 selftask：共享队列 + 原子 Claim

```json
// data/dev-tasks.json
{
  "tasks": [
    {
      "id": "task-001",
      "description": "创建 hello.txt 文件",
      "status": "pending",          // pending → working → done/failed
      "created_at": "2026-02-23T09:40:00Z",
      "worker_id": null,            // claim 时填入
      "started_at": null,
      "completed_at": null
    }
  ]
}
```

**机制**：
- 原子 claim：`load → find first pending → mark working → save`
- 状态机：`pending → working → done | failed`
- Worker 退出后由 Manager 重启
- 任务失败可被重新领取

### 5.2 长时间运行：Feature List 接力

```json
// feature_list.json（Initializer 生成，Coding Agent 推进）
[
  {
    "category": "functional",
    "description": "User can log in with email/password",
    "steps": ["Navigate to /login", "Enter credentials", "Submit", "Verify redirect"],
    "passes": false              // 唯一可修改字段：false → true
  }
]
```

**机制**：
- Initializer Agent 一次性生成完整特性列表（25+ 条）
- Coding Agent 每个 Session 选择最高优先级未完成特性
- 只允许修改 `passes` 字段（不可变架构）
- 进度通过 `claude-progress.txt` 传递上下文

### 5.3 自测试迭代：YAML 配置驱动

```yaml
# scenarios/basic_task.yaml
id: basic_task
name: 基本任务场景
goal: Complete standard consultation
persona_ids: [novice, expert, impatient]
max_turns: 8
success_keywords: [完成, 已处理]
script:
  - "I want to complete a basic task"
  - "Please help me"
```

**机制**：
- YAML 定义测试维度（Persona、Scenario）
- 运行时自动组合
- 无需人工管理任务状态
- 结果通过 JSON 记录永久存档

### 5.4 对比

| 维度 | selftask | 长时间运行 | 自测试迭代 |
|------|----------|-----------|-----------|
| **任务定义** | 自由文本描述 | 结构化特性 + 测试步骤 | YAML 场景 + 脚本 |
| **任务来源** | 人工添加到队列 | Agent 自动生成 | 配置文件预定义 |
| **任务状态** | 4 态（pending/working/done/failed） | 2 态（passes: true/false） | 无状态（每次全量运行） |
| **并发安全** | 原子 claim | 单 Agent 无并发问题 | 单进程无并发问题 |
| **任务复杂度** | 低（单一操作） | 高（需浏览器测试） | 中（多轮对话） |
| **可追溯性** | task ID + worker ID + 时间戳 | git log + progress file | run_id + record_id |

## 6. 上下文管理策略对比

这是三个项目最核心的差异之一。

### 6.1 selftask：无上下文（Stateless Workers）

```
Worker 启动 → 读 CLAUDE.md → 领取任务 → 执行 → 退出
                  ↑
            每次全新上下文，不记忆历史
```

- 每个 Worker 是全新的 Claude Code 进程
- CLAUDE.md 提供固定指令（不变的"工作手册"）
- 不需要跨任务传递上下文
- **设计哲学**：任务足够小，不需要上下文

### 6.2 长时间运行：多层上下文恢复

```
Session N 启动
    ├── 读取 feature_list.json     → 知道「要做什么」
    ├── 读取 claude-progress.txt   → 知道「做到哪了」
    ├── 读取 git log               → 知道「做了什么」
    ├── 读取 Knowledge Base        → 知道「踩过什么坑」
    └── Observer 注入告警          → 实时「避免重蹈覆辙」
```

- **五层上下文传递**：feature_list / progress / git / KB / Observer
- 每个 Session 开头执行「上下文恢复」步骤
- 10 步标准流程确保不遗漏
- **设计哲学**：用文件系统模拟"长期记忆"

### 6.3 自测试迭代：配置即上下文

```
sim-dev run
    ├── 读取 sim-dev.yaml          → 知道「测什么」
    ├── 读取 personas/*.yaml       → 知道「谁来测」
    ├── 读取 scenarios/*.yaml      → 知道「测什么场景」
    └── 读取 assertions/*.py       → 知道「怎么判断」
```

- 上下文完全由配置文件决定
- 无需跨 Session 传递状态
- 历史通过 records/ 和 reports/ 保存
- **设计哲学**：配置文件 = 测试规范，运行结果 = 测试报告

### 6.4 上下文策略对比

| 维度 | selftask | 长时间运行 | 自测试迭代 |
|------|----------|-----------|-----------|
| **上下文模型** | 无状态 | 有状态（文件传递） | 无状态（配置驱动） |
| **恢复机制** | 不需要 | 多层文件读取 | 每次全量配置加载 |
| **记忆持久性** | PROGRESS.md（手动） | feature_list + KB（自动） | records/（自动） |
| **上下文窗口压力** | 低（单任务） | 高（需恢复大量历史） | 低（配置固定） |
| **信息丢失风险** | 低（无需传递） | 中（依赖文件完整性） | 低（无需传递） |

## 7. 质量保障机制对比

### 7.1 selftask：最小化质量保障

```
执行任务 → (可选) 运行测试 → 测试失败 → 重试一次 → 仍失败 → 标记 failed
```

- **测试运行**：`npm test || pytest || go test || echo "无测试"`（万能降级）
- **失败策略**：重试一次，仍失败则放弃
- **质量门禁**：无（依赖 Claude Code 自身能力）
- **代码审查**：无自动审查

### 7.2 长时间运行：多层防御 + 自我改进

```
Layer 1: Security Hook（阻止危险命令）
    ↓
Layer 2: Observer Hook（检测运行时问题）
    ↓
Layer 3: Coding Prompt 7 条验证规则
    ↓
Layer 4: 浏览器自动化测试（真实 UI 验证）
    ↓
Layer 5: 回归验证（每个 Session 先验证旧功能）
```

- **14 种已验证问题模式** 的自动检测
- **自动修复**：Singleton Lock 清理、端口释放等
- **实时告警注入**：通过 stdout 影响 Agent 行为
- **回归防护**：新 Session 必须先验证旧功能
- **83% 自愈率**：30 个问题中 25 个自动解决

### 7.3 自测试迭代：结构化双门评估

```
Layer 1: Hard Gate（断言插件）
    ├── no_crash: 系统不崩溃
    ├── response_format: 响应不为空
    └── (自定义断言)
    ↓ 必须全部通过
Layer 2: Soft Gate（体验评分）
    ├── 目标达成率 (>80%)
    ├── 平均满意度 (>3.5/5)
    └── 困惑热点分析
    ↓ 指导改进方向
Layer 3: 版本对比
    └── compare: 量化版本间改进
```

- **插件化断言**：动态加载 Python 模块，零核心耦合
- **多维评估**：成功率 × 满意度 × 步数 × Token 消耗
- **结构化报告**：Markdown 格式，含具体修复建议
- **CI/CD 集成**：`--fail-on-hard-gate` 返回非零退出码

## 9. 安全机制对比

### 9.1 selftask：最小安全

```
Claude Code --dangerously-skip-permissions
```

- 完全跳过权限检查（为了自动化）
- 依赖 Git Worktree 隔离（物理隔离）
- .env 文件不提交（基本安全）
- 无命令白名单

### 9.2 长时间运行：三层安全架构

```
Layer 1: 命令提取
    └── Quote-aware parser（正确处理引号内分号）

Layer 2: 白名单检查
    └── 60+ 允许的命令（ls, git, npm, python3, etc.）
    └── 未知命令 → 阻止

Layer 3: 特殊命令验证
    ├── pkill: 只允许杀开发进程（node, npm, python, uvicorn）
    ├── chmod: 只允许 +x
    └── rm: 阻止 rm -rf / 等危险模式
```

- **async hook**：每次 Bash 调用前验证
- **完整测试套件**：`test_security.py` 覆盖各种边界情况
- **沙箱关闭但有白名单防护**：攻防平衡

### 9.3 自测试迭代：无额外安全

- 作为 CLI 工具，不涉及 Agent 执行代码
- 目标系统通过 Adapter 隔离
- Mock Adapter 完全无副作用
- HTTP/CLI Adapter 有超时保护（15秒）

|------|----------|-----------|-----------|
| **安全模型** | Worktree 隔离 | 白名单 Hook | Adapter 隔离 |
| **权限控制** | 跳过（自动化需要） | 白名单 + 特殊验证 | N/A |
| **危险命令防护** | ❌ | ✅ (三层) | N/A |
